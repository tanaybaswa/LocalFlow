Summary

On an Apple M5 Max (36 GB unified memory), OpenAI Whisper large-v3 via whisper.cpp sustains approximately 12–17× realtime across lecture, tutorial, and podcast workloads. A 15-minute interview segment finished in ~62 s wall time. For product decisions, transcription latency is no longer the binding constraint — quality ceiling and UX (record → transcript) matter more than trading down to a smaller model.

Decision framing: Keep large-v3 as the default local engine. Treat turbo / quantized variants as optional fast paths, not as requirements to unlock interactive use.

Model selection

Goal: maximize transcription quality for local, on-device use without cloud dependency. Hardware (M5 Max, 36 GB, Metal 4, 32 GPU cores) removes memory as a practical limit for the Whisper family.

Chosen model

Upstream openai/whisper-large-v3 (~1.54B params)

Why this model Highest accuracy tier in Whisper; fits easily in 36 GB; speed still ≫ realtime on Metal

Analysis

Throughput is product-ready

Interactive transcription (record 1–3 minutes → wait for text) should feel near-instant after stop: expected wall times of ~5–15 s for typical utterances on this stack. Batch jobs (podcast episode, lecture) complete at roughly one minute of compute per ~15 minutes of audio.

Content type affects × realtime

  • Explanatory lecture (3b1b): highest throughput (~15–17.5×) — clearer pacing, fewer overlaps.
  • Dense interview (Lex): ~14.6× — still strong; more continuous speech and turn-taking.
  • Fast tutorial (Fireship): lowest of the set (~11.6×) — rapid delivery increases decode work per second of audio.

What this does not measure

  • Word error rate (WER) / proper ASR accuracy benchmarks
  • Cold-start vs warm model-resident latency (each run spawned a fresh whisper-cli)
  • Streaming / partial hypotheses (batch file mode only)
  • Speaker diarization, timestamps quality, or non-English
  • Concurrent jobs or thermal throttling under sustained load
Wall times include model load. A long-lived server process (whisper-server) or in-process Metal session would improve first-token / short-clip latency further. That is the main optimization lever if a native app feels “slow on short recordings.”

Conclusions & recommendations

  1. Keep ggml-large-v3.bin as default. Hardware headroom and measured throughput justify the quality ceiling.
  2. Do not block the next app on a smaller model. Speed is already past the interactive threshold for batch STT.
  3. Next app focus should be UX and integration: record → transcript → edit/copy/export; optional always-on whisper-server to remove per-request load cost.
  4. Optional follow-ups (not blockers): WER on a labeled set; A/B vs large-v3-turbo; streaming mic path; diarization for meetings.

Go for building the next product surface on this stack. Treat model choice as settled unless accuracy tests later prove otherwise.

  --model models/ggml-large-v3.bin \
  --file samples/bench/lex_15min.wav \
  --language en \
  --threads 8 \
  --no-prints \
  --output-txt \
  --output-file outputs/bench/lex_15min
